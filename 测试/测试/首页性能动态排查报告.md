## 2. 本轮已实施优化

### 2.1 HomeLayerFlow 拆掉滚动驱动布局

- 保留海层识别、当前区块切换和导航同步。
- `--home-layer-progress` 继续保留，但只再驱动 transform / opacity 一类视觉变量。
- 去掉 `featured-destinations`、`why-yanqi` 上跟随滚动不断变化的 `margin-bottom` / `padding-bottom`。
- `HomeLayerFlow` 的滚动进度改为更粗粒度分桶，减少持续 style write。
- `ResizeObserver -> requestMeasure()` 改成只在 section 高度出现明显变化时才触发。

### 2.2 Hero BambooScroll 增加页面滚动 settling 窗口

- 新增首页内部 `scrollSettling` 状态源，只服务首页高频模块，不做机器档位持久化。
- 快速滚轮 / 大幅滚动时，hero 竹签卡片优先停掉：
  - auto step
  - inertia
  - hover 命中同步
  - physics / wobble 帧循环
- 首轮 `measure()`、`centerOnSpotId()`、`syncMotionLiteState()` 合并到一次稳定后的初始化批次。
- `bamboo-cards-wrapper`、`hero-bamboo-cards-wrapper` 增加局部 `contain`，缩小滚动时的重绘/重排影响范围。

### 2.3 Curated / DiveMatch 避免滚动途中继续长 DOM

- `CuratedWatersStage.runInitialHydrationStep()` 去掉中途两次 `updateOnly` 刷新，只保留最终一次尾刷。
- `DiveMatchStage.hydrateMatchSurfaceCards()` 不再按每帧 2 张卡片分批插入。
- 改成：
  - 首次离屏初始化：空闲时一次性补齐卡片
  - 当前区块已稳定可见：下一帧直接一次性补齐
  - 完成后只做一次尾部 measure
- `dive-match-card-grid` 增加稳定 `min-height`，减少 skeleton -> 实卡阶段的布局位移。

### 2.4 DepthManager 滚动期进一步粗化

- 首页快速滚动时，深度计 fast path 现在也会识别 `scrollSettling`。
- 快速滚动期：
  - 更粗频率排队 `queuePageScrollDepthUpdate()`
  - 更早 snap / 更快收敛
  - overlay / gauge / marker 继续走粗粒度渲染
- 停稳后自动回到完整渲染。

### 2.5 延迟模块更早在空闲期启动

- `#featured-destinations` 和 `#dive-match` 现在都会在空闲窗口尽量提前 bootstrap。
- 目标是把“用户已经开始滚了，模块才初始化”的情况尽量前移到首屏空闲期。

## 3. 实测方式

- 本地静态服务：`http://127.0.0.1:8000/site/home.html`
- Chrome 147，CDP 动态采样
- 关闭缓存，绕过 Service Worker
- 两组场景：
  1. `lowpc-real`
  2. `lowpc-cpu4`

脚本里的滚动动作不是连续丝滑滚轮，而是 6 段 `instant scrollTo` 跳步下滚，所以这份数据更偏“滚动链路抗冲击能力”，很适合看掉帧、重排和长任务。

## 5. 结果解读

### 5.1 已经确认有效的部分

- `lowpc-real` 这组改善是明确的，尤其是滚动后的样式重算和总任务时长，说明这次“滚动驱动布局 -> 纯视觉变量”和“hero 滚动 settling 停机窗”是有效的。
- `lowpc-cpu4` 下 Long Task 数从 `32` 压到 `26`，说明高压场景里连续冻结段已经被打散一部分。
- `TaskDuration`、`LayoutDuration` 也有小幅下降，说明 hydration 尾刷和深度计粗化没有白做。

### 5.2 还没打透的部分

- 当前最顽固的问题是 `CLS` 仍然稳定在 `0.14+`。
  - 这说明首页还存在一段“加载后早期可见布局位移”，不是单纯滚动期问题。
  - 这部分不在本轮主要改动命中面里，所以数值改善有限。
- `lowpc-cpu4` 的 `RecalcStyleDuration` 还在 `3.874s`，说明极端慢 CPU 下，首页滚动后依然有比较重的整页样式重算。
  - 本轮已经去掉了一部分连续布局写入。
  - 但首页整体 DOM 规模、filter/backdrop/glass 系统、以及多模块 class/状态联动，仍然会放大样式计算成本。
- `homePerformanceMode` 仍然是 `full`。
  - 这是符合当前约束的结果，因为这轮明确不做长期设备档位降级。
  - 代价就是慢机器仍然要扛下完整静止态视觉。

## 6. 现阶段结论

这轮优化已经把“滚动掉帧最直接的几条主链路”明显压下去了一截，尤其是：

- `HomeLayerFlow` 不再边滚边改布局
- Hero BambooScroll 不再在页面快速滚动时继续后台抢主线程
- DiveMatch 不再边滚边多批次补卡并重复刷新
- DepthManager 在滚动爆发期会更粗频率地工作

但这还不是终局版，当前首页还有两块残余热点：

1. 首屏后 1 秒左右的可见布局位移，导致 `CLS` 仍在 `0.14+`
2. 极慢 CPU 下整页样式系统仍然偏重，导致 `RecalcStyleDuration` 没完全压下来

## 7. 下一步建议

如果继续按“不能长期降静止视觉、只做桌面端、优先压滚动掉帧”的方向推进，下一轮最值得做的是：

1. 逐项定位 `CLS` 来源
   - 用 Layout Shift source 再抓一轮，确认是不是 hero 醒场、导航高度、头像回流、还是首屏图片区在推位。
2. 收首页静态态之外的 class 联动范围
   - 特别是滚动后仍会被频繁命中的大容器 filter / glow / backdrop 组合。
3. 把首页部分下面板改成“固定壳 + 内容替换”
   - 继续减少进入区块后整块样式树重新参与计算的机会。
