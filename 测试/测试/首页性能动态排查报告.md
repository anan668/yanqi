# 盐憩首页性能动态排查报告

生成时间：2026-04-16

## 1. 排查目标

本次排查针对 `site/home.html` 在低性能电脑上的真实运行表现，重点不是只看静态代码，而是结合动态采样确认：

- 首屏为什么会在慢机器上变重
- 滚动后为什么更容易掉帧
- 为什么会出现明显布局位移
- 为什么不同机器表现差异很大

## 2. 本次测试方式

测试脚本：

- `tools/qa/home-cdp-profile.ps1`

动态调试方式：

- 本地静态服务：`http://127.0.0.1:8000/site/home.html`
- Chrome 147 + CDP 动态采样
- 已关闭缓存、绕过 Service Worker
- 采样包含：加载阶段、自动滚动阶段、Long Task、CLS、Trace

本次实际跑了两组：

1. `lowpc-real`
   当前这台低性能电脑上的真实采样，不额外做 CPU 节流。
2. `lowpc-cpu4`
   在同一台机器上加 `4x` CPU 节流，模拟更差的低端压力场景。

结果文件：

- `tools/qa/out/home-cdp-profile/lowpc-real-summary.json`
- `tools/qa/out/home-cdp-profile/lowpc-real-trace.json`
- `tools/qa/out/home-cdp-profile/lowpc-cpu4-summary.json`
- `tools/qa/out/home-cdp-profile/lowpc-cpu4-trace.json`

## 3. 关键结论

### 总结

- 首页现在的主要瓶颈不是单一慢请求，而是“首屏初始化过满 + 延迟区块继续补 DOM + 滚动期持续动画 + 多次统一测量”叠在一起。
- 这台机器在页面里仍被判定为 `homePerformanceMode = full`，没有切到 `balanced` / `lite`，所以很多高成本动画和首屏行为没有自动降级。
- 真机低性能场景下，`CLS` 稳定在 `0.15` 左右，已经明显高于良好阈值 `0.1`。
- 一旦再加 `4x` CPU 节流，首屏和滚动阶段都会出现成串的超长任务，说明当前实现对慢机器不够稳。

### 为什么不同机器差异大

当前性能档位判断来自 `site/js/home.js:697` 的 `resolveHomePerformanceProfile()`。

这次动态采样里浏览器上下文是：

- `deviceMemory = 8`
- `hardwareConcurrency = 8`
- `viewport = 1440x1200`
- `coarsePointer = true`
- `finePointer = true`

因此最终仍然判成：

- `mode = full`
- `lowEnd = false`
- `lite = false`

也就是说，这台用户主观上已经觉得“低性能”的电脑，在当前规则里并不会触发降载，所以会直接吃到完整动画、完整首屏初始化和完整滚动联动。

### 真实低性能机器 `lowpc-real`

关键结果：

- `homePerformanceMode = full`
- 首屏 DOM 节点：`967`
- 首屏图片数：`47`
- 首屏脚本数：`10`
- 首屏资源总量：约 `3.60 MB`
- 滚动后：
  - `RecalcStyleDuration = 1.633s`
  - `TaskDuration = 3.893s`
  - `longTaskCount = 8`

真实机器上最明显的问题不是首屏白屏，而是：

- 首屏后还有明显位移
- 用户开始滚动后，主线程继续被动画帧和布局计算占住
- 慢机器上“感觉卡”的来源更多在滚动和持续联动阶段

### 压力场景 `lowpc-cpu4`

关键结果：

- `homePerformanceMode = full`
- 首屏资源总量：约 `2.65 MB`
- 加载后：
  - `FCP = 1164ms`
  - `LCP = 1380ms`
  - `load = 1565.6ms`
- 滚动后：
  - `TaskDuration = 9.198s`
  - `RecalcStyleDuration = 3.779s`
  - `LayoutDuration = 1.933s`
  - `longTaskCount = 32`

滚动后最重长任务：

- `967ms`
- `522ms`
- `490ms`
- `477ms`
- `453ms`
- `395ms`
- `387ms`
- `378ms`

结论非常明确：

- 当前首页在更慢 CPU 下不是“稍微变慢”，而是会进入连续长任务状态。
- 一旦机器更差、后台更忙、功耗受限或浏览器有额外压力，滚动和切换体验会明显失稳。

## 6. 当前确认的问题

### 问题 A：低性能机器没有触发降级档位

级别：高

现象：

- 当前机器主观体验已经偏慢，但页面依旧进入 `full` 档位。

影响：

- 不会自动关闭更重的首屏行为
- 不会自动压缩滚动阶段动画密度
- 不会自动减轻首屏卡片和交互链路

相关位置：

- `site/js/home.js:697`
- `site/js/home.js:724`

### 问题 B：首屏初始化链路过满

级别：高

现象：

- `DOMContentLoaded` 附近同时启动了过多模块
- `DepthManager` 还在 `DOMContentLoaded` 前就已经运行

影响：

- 首屏长任务明显
- 慢机器上脚本解析、DOM 构建和测量容易叠加

相关位置：

- `site/home.html:393`
- `site/js/home.js:6543`
- `site/js/home.js:6561`
- `site/js/home.js:6568`
- `site/js/depth-manager.js:3833`

### 问题 C：BambooScroll 首屏建树和逐帧动画成本高

级别：高

现象：

- 首屏会生成多套卡片克隆
- 生成后立即同步测量
- 滚动后长期维持 `requestAnimationFrame`

影响：

- 首屏容易出现 DOM 构建和布局热点
- 滚动期容易出现持续 `FunctionCall`
- 低性能机器更容易掉帧

相关位置：

- `site/js/home.js:2088`
- `site/js/home.js:2124`
- `site/js/home.js:2212`
- `site/js/home.js:2763`
- `site/js/home.js:2811`
- `site/js/home.js:2903`
- `site/js/home.js:3277`

### 问题 D：HomeLayerFlow 会把滚动转成布局写入

级别：高

现象：

- 滚动时持续写 `class` 和 `--home-layer-progress`
- CSS 直接把这些进度映射到 `padding-bottom` / `margin-bottom`
- 还叠加了 `ResizeObserver -> requestMeasure()`

影响：

- 这是最像 `RecalcStyle / Layout` 放大器的一段
- 慢机器滚动时更容易出现布局抖动

相关位置：

- `site/js/home.js:6474`
- `site/js/home.js:6532`
- `site/css/home.css:6000`
- `site/css/home.css:6182`

### 问题 E：CLS 主要来自延迟挂载与分批补内容

级别：高

现象：

- `Curated Waters` 初始留空，后续分阶段补 rail 和主卡
- `Dive Match` 先 skeleton，再分批 `insertAdjacentHTML` 补卡
- 每次补内容都配合 `requestMeasure()`

影响：

- 用户滚动时下方区块才继续长出来
- 慢机器上更容易出现持续位移
- 这就是 `CLS≈0.15` 的主要来源之一

相关位置：

- `site/js/home.js:3384`
- `site/js/home.js:3525`
- `site/js/home.js:4936`
- `site/js/home.js:5004`
- `site/js/home.js:401`
- `site/js/home.js:6572`

### 问题 F：首页图片仍偏重，且首屏优先级过激进

级别：中高

现象：

- 最重图片仍在 `300KB~500KB+`
- 首屏中心卡是 `loading="eager" fetchpriority="high"`
- 相邻卡也会 `loading="eager"`
- 克隆卡让同一批图片多次出现在 DOM 中

影响：

- 会拖慢页面稳定下来的时刻

相关位置：

### 问题 G：HomeSeaGuide 可见态更偏 GPU 重

级别：中

现象：

- 滚动后持续参与 section 高亮更新
- 面板可见态用了 `backdrop-filter`
- 还有常驻漂浮动画

影响：

- JS 不是最重，但在低性能电脑上容易造成“发虚”和“拖影感”

## 7. 分 Agent 排查结果

### Agent 1：首屏初始化 / 长任务

结论：

- 长任务主要来自两条链路叠加：
  - `DepthManager` 在脚本执行期就开始创建深度计、绑定深度系统
  - `DOMContentLoaded` 后首页又立刻连续初始化多个模块
- 最重热点不是单点，而是：
  - `BambooScroll.render()`
  - `BambooScroll.measure()`
  - `homeViewportCoordinator.requestMeasure()`
  - 首屏阶段过早接入 `HomeSeaGuide` / `HomeLayerFlow` / `ResizeObserver`

优先关注：

- `site/js/home.js:2124`
- `site/js/home.js:2212`
- `site/js/home.js:6543`
- `site/js/depth-manager.js:1545`
- `site/js/depth-manager.js:3833`

### Agent 2：滚动 / 动画 / GPU

结论：

- 滚动后持续占用主线程和 GPU 的三条主链路：
  - `BambooScroll`
  - `DepthManager`
  - `HomeLayerFlow`
- 其中最像布局放大器的是 `HomeLayerFlow`
- `HomeSeaGuide` 的 JS 不算最重，但玻璃模糊和漂浮动画对低性能机器不友好

优先关注：

- `site/js/home.js:2811`
- `site/js/home.js:2903`
- `site/js/home.js:6474`
- `site/js/home.js:6532`
- `site/js/depth-manager.js:2777`
- `site/js/depth-manager.js:2943`

### Agent 3：图片 / 资源 / CLS

结论：

- 当前 `CLS≈0.15` 更像“延迟补内容 + 多次重测”问题，不是单张图片缺少宽高的问题
- 首屏图片大是放大器，但真正让版面位移的是：
  - `Curated Waters` 延迟初始化
  - `Dive Match` skeleton 后分批补卡
  - deferred bootstrap 让区块在滚动途中继续长出来

优先关注：

- `site/js/home.js:3384`
- `site/js/home.js:3525`
- `site/js/home.js:4936`
- `site/js/home.js:5004`
- `site/js/home.js:6572`

## 8. 当前建议的优化优先级

如果下一步进入真正优化，建议按下面顺序推进：

1. 先让低性能机器能够正确降级，不要继续走 `full` 档
2. 收掉 `HomeLayerFlow` 的布局写入和 `ResizeObserver` 反馈链
3. 压缩 `BambooScroll` 的首屏建树体量和滚动期逐帧计算
4. 把 `Dive Match` 和 `Curated Waters` 从“延迟补布局”改成“先稳定布局，再延迟特效”
5. 降低首页图片体积和首屏 eager 图片数量
6. 缩短 `DepthManager` 滚动后的自驱动收敛时间
7. 最后处理 `HomeSeaGuide` 的玻璃模糊和漂浮动画
