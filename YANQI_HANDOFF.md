## 1. 项目概述

项目名称：盐憩  
项目类型：潜水旅游主题网页

这是一个围绕“潜水旅行体验”展开的品牌化网页项目。  
它的目标不是做一个普通旅游信息站，也不是做一个工具型预订平台，而是通过页面结构、深度计、模块叙事、浮层系统和潜浮式动画，把“浏览一片海、理解一片海、把一片海排进行程”做成一套连续体验。

项目整体气质已经明确：

- 深海感
- 安静感
- 高级感
- 舒缓、沉浸
- 有呼吸感
- 不追求炫技

品牌语义也已经固定：

- “盐”代表海、海水、潜水环境
- “憩”代表停驻、慢下来、在海里留出空间

所以这个项目始终不是“功能先行”，而是“先建立世界观，再把功能收进这套世界观里”。

## 2. 统一设计原则

后续所有继续优化，都必须遵守这组原则。

### 2.1 必须保持的方向

- 潜水主题
- 深海感
- 安静感
- 高级感
- 低饱和海洋配色
- 玻璃感 / 雾感
- 微发光
- 舒缓、沉浸
- 有呼吸感

### 2.2 明确不要的方向

- 普通旅游网站感
- 电商感
- 后台系统感
- 工具面板感
- 普通轮播 / 普通搜索栏 / 普通锚点目录
- 夸张科技风
- 强 3D、强弹簧、强炫光

### 2.3 统一判断标准

如果某个模块看起来像：

- 普通卡片列表
- 普通搜索框
- 普通下拉框
- 普通页脚
- 普通推荐区

那通常都要继续重做，直到它能回到“盐憩”的语境里。

## 3. 页面层级关系

这个项目的页面关系不是平级切换，而是海层关系。

### 3.1 `home.html`

首页是浅层入口。  
它更亮、更开放，更像海面入口。

定位：

- 看海
- 被海吸引
- 进入世界观
- 逐步理解不同海域与不同用户之间的关系

### 3.2 `trip.html`

行程页是更深一层的规划空间。  
它比首页更安静、更私密、更收束。

定位：

- 把海收进行程
- 慢慢安排行程
- 不像填写表单，而像在一层更深的海图上做决定

### 3.3 `detail.html`

详情页是进入某一片具体海域。  
不是景点详情页，而是“进入这一片海”。

定位：

- 看这片海的情绪
- 看水下结构
- 看套餐是否适合自己
- 看别人如何在这片海停驻
- 再从这片海延展到另一片海

### 3.4 页面切换原则

- `home -> trip`：像继续下潜到更深一层
- `trip -> home`：像从更深处缓慢上浮回入口
- `detail -> detail`：像在相邻海域之间潜游和平移

原则上不要把页面切换做成普通网页跳转。

## 4. 当前已明确的模块设计方向

下面按页面整理当前已经基本定下来的方向。后续继续做时，优先在这些方向上细化，不要轻易推翻世界观。

### 4.1 `home.html`

#### Hero

- 首屏不是搜索页
- 是品牌入口
- 标题主线是“下一次下潜，从哪里开始？”
- 用户进入首页时，应该感觉自己刚开始下潜

#### 今日海域 / Hero 区内容

- 目前 Hero 下方已经不是普通搜索栏
- 更偏品牌化内容卡和热门潜点舞台
- 需要保持“海面入口”的感觉

#### Curated Waters / 精选目的地

- 已经从普通网格卡改成更舞台化的展示
- 不再是普通旅游列表
- 当前是更有层次的目的地陈列方式
- 需要继续保持“不是更多的海，而是更适合你的那一片”的逻辑

#### Dive Match / 按能力与需求匹配海域

- 已经插入到首页结构中
- 位置在 `featured-destinations` 下方，`why-yanqi` 上方
- 不是筛选器，而是“按下潜能力与需求，找到更适合你的海”
- 目前已支持分类切换、推荐内容切换、和深度计联动

#### 盐憩故事

- 已经从普通卖点区改成品牌叙事区
- 不是功能说明，而是品牌存在理由与海洋叙事
- 要继续保持“不是平台卖点，而是品牌世界观”

#### Footer

- 首页 footer 已经重做成品牌收束区
- 不是普通三栏页脚
- 更像首页体验结束后的品牌停驻区

#### 首页海图导览

- 已经做了 `sea-guide-home`
- 替代旧的普通 back-to-top
- 可以跳到首页关键 section
- 外层点击穿透已修，避免遮挡内容

### 4.2 `trip.html`

#### trip hero

- 已重做，不再使用右侧 `trip-hero-note`
- 不再是“左边标题 + 右边状态卡”
- 现在是统一的大舞台：标题、说明文案、Planner Desk 一体化
- 视觉上像更深一层的海图工作台

#### Planner Desk

- 已从普通搜索栏重做成深海玻璃控制台
- 海域不能自由输入，只能固定选项选择
- 同行人数也不再是普通 select
- 出发时间已改成自定义潮汐日历浮层
- 摘要区会实时回写用户选择

#### 浮层系统

- 海域选择、同行人数、出发时间都已改成独立浮层系统
- 不再使用字段内部挤出菜单
- 浮层需要完整显示，不依赖页面滚动配合
- 同时只允许一个浮层打开

#### trip-prep / 出发前准备

- 这块已经不是静态说明卡
- 已升级成更完整的准备系统方向
- 但后续仍有继续深化空间

#### trip footer

- 目前风格已经和全站统一
- 但后续可继续与首页 / 详情页收束区做更强的一致性细调

### 4.3 `detail.html`

#### 英雄区

- 不是普通 banner
- 每个潜点有自己的英雄图气质
- 马布岛等特殊潜点使用专属图与专属文案
- 有缓慢漂移 / 呼吸感

#### 预订区

- 已按能力匹配逻辑重构
- 分成休闲套餐和进阶套餐
- 先看详情，再决定是否预订
- `booking-match` 标签可跳回首页 `Dive Match` 对应分类

#### 评论区

- 已不是简单文字评论
- 设计方向是“旅行回声卡片”
- 有照片、总评、潜水/住宿/饮食/风景四维信息
- 有评论详情弹窗
- 图片支持放大查看

#### 地图区

- 已不再是普通地图占位
- 已升级成三态 `Sea Atlas / 海域定位台`
- 包含：
  - 海域位置
  - 到达方式
  - 水下结构

#### “你可能还喜欢”

- 已重做成更舞台化的“还可以继续看的海”
- 不再是普通推荐卡列表
- 逻辑上是：从当前这片海，再望向另一片海
- 有主卡、相邻海域卡、横向潜游式切换

#### detail footer

- 已重做
- 不是普通页脚
- 是这片海的余波与收束区
- 更适合作为详情页情绪上的结尾

#### detail 海图导览

- 已实现悬浮式海图导览系统
- 可跳转：
  - 回到海面
  - 先看这片海
  - 看行程与套餐
  - 找到这片海的位置
  - 听听别人的回声
  - 还可以继续看的海
  - 把这一潜慢慢收住（页脚）
- 已修复遮挡问题：外层容器不再拦截页面点击

## 5. 当前已定的重要交互和视觉规则

以下规则现在已经是项目的“核心交互协议”，后续不要随便改回普通网页逻辑。

### 5.1 深度计系统

- 深度计已经不是装饰，而是页面层级语言
- 当前逻辑是按“海层”组织页面
- 刻度已升级为每米细刻度
- 每 5 米中刻度
- 每 10 米主刻度与数字
- 深度变化必须平滑，不要突跳

### 5.2 首页滚动深度联动

- 首页不同 section 对应不同海层
- 往下滚动时深度计应逐步变深
- `Dive Match` 模块比 `Curated Waters` 更深
- `盐憩故事` 和 footer 再更深

### 5.3 模块切换时的深度计微调

- `Dive Match` 内部分类切换会引起小幅深度变化
- 这种变化不是大跳，而是柔和微调

### 5.4 页面切换规则

- `home <-> trip` 走潜浮式过渡
- `detail -> detail` 走横向潜游式过渡
- 页面切换不能只是普通 fade
- 过渡层应继续服务于“进入另一层海”

### 5.5 浮层系统规则

- 海域 / 人数 / 日期浮层都是独立浮层
- 不依赖字段内部展开
- 打开时从字段附近浮出
- 页面不需要配合滚动
- 点击外部关闭
- `ESC` 关闭
- 同一时刻只开一个

### 5.6 `booking-match` 联动规则

- 详情页中的 `booking-match` 标签必须能跳回首页 `Dive Match`
- 跳回时不仅要滚到 `#dive-match`
- 还要自动激活对应分类
- 这个逻辑已经建立，后续不要退回成普通链接

### 5.7 海图导览规则

- 首页和详情页都已接入海图导览
- 目的不是“回到顶部按钮”，而是“海图中的快速切换”
- 点击后滚动要走舒缓、有路径感的滚动
- 当前所在 section 需要轻微高亮反馈

### 5.8 评论 / 套餐 / 相关推荐的弹层与切换

- 评论详情弹窗要始终视口居中
- 套餐详情要先展开，再让用户决定是否预订
- 推荐区切换不能像普通轮播
- 需要保持“当前海退后，另一片海靠近”的逻辑

## 6. 已知问题与解决方向

下面是这个项目曾经遇到过、或者后续继续修改时很容易再次踩到的问题。

### 6.1 浮层被裁切 / 乱飘

曾经的问题：

- 海域选择
- 同行人数
- 日期面板

会被父容器裁切，或随着滚动显得不稳定。

解决方向已经明确：

- 浮层与字段分离
- 浮层挂高层容器
- 根据字段位置动态定位
- 不依赖普通 dropdown 模式

### 6.2 评论弹窗不居中

曾经的问题：

- 浮层在布局容器内
- 受 transform / overflow 影响
- fixed 定位失效

解决方向：

- 提升到 `body` 顶层
- 统一做视口级 modal
- 打开时锁定背景滚动

### 6.3 原生日历太普通

曾经的问题：

- 原生 `date input` 太像系统控件
- 不能融入盐憩的视觉语言

解决方向：

- 改成自定义潮汐日历浮层
- 把“选日期”改成“选出发窗口”

### 6.4 trip 右侧 note 卡太像 dashboard

曾经的问题：

- trip 首屏太像说明卡 + 状态面板
- 和“深海规划台”定位冲突

解决方向：

- 移除 `trip-hero-note`
- 把 Hero 和 Planner Desk 融成一个更完整的大舞台

### 6.5 detail 推荐区太普通

曾经的问题：

- 推荐区像低级旅游卡片列表
- 破坏详情页结尾情绪

解决方向：

- 改成“还可以继续看的海”
- 改成主卡 + 相邻海域结构
- 切换逻辑走横向潜游式过渡

### 6.6 地图区太普通

曾经的问题：

- 普通地图无法体现潜水信息层次

解决方向：

- 升级成 `Sea Atlas`
- 强化海域位置 / 到达方式 / 水下结构三态

### 6.7 深度计曾经只是装饰

曾经的问题：

- 深度计只是视觉元素
- 没有真正参与页面逻辑

解决方向：

- 细化每米变化
- 让它跟滚动、模块切换、页面切换联动
- 让它成为整站空间语言

### 6.8 悬浮海图导览遮挡内容

曾经的问题：

- 首页 / 详情页海图导览外层会挡住下方按钮
- 典型表现是滚动后套餐卡 `查看详情` 点不动

解决方向已经落地：

- 外层容器保持点击穿透
- 只让 trigger 和 panel 接收点击

## 8. 给下一个账号的工作说明

后续继续帮这个项目时，请直接遵守以下方式。

### 8.1 输出方式

优先输出：

- 可以直接交给 Codex 执行的纯文字修改方案
- 或直接在代码里落地修改

不要退回到非常泛泛的建议，不要只给“可以这样做”的空思路。

### 8.2 审美和设计基准

不要退回普通网页设计思路。  
所有后续建议必须继续围绕：

- 潜水
- 深海
- 停驻
- 海层
- 安静、高级、舒缓

展开。

### 8.3 设计判断标准

如果一个模块看起来像：

- 普通旅游平台
- 普通推荐区
- 普通 dropdown
- 普通页脚
- 普通 dashboard

那通常说明还没到位，需要继续往“盐憩”的世界观里收。

### 8.4 后续继续修改时的优先原则

1. 先保住世界观统一  
2. 再细化模块交互  
3. 再增加功能  

不要为了加功能破坏原本已经建立起来的海层语言。

### 8.5 如果下一个账号要继续做

建议优先做这类任务：

- 继续优化已有模块，而不是新增杂乱功能
- 优先把普通控件继续品牌化
- 优先让页面之间、模块之间更统一
- 优先处理“哪里还像普通网页”这个问题

一句话给下一个账号的工作说明：

继续这个项目时，请始终把它当作一个“由不同深度、不同节奏、不同海域组成的网站”，而不是几个页面和几个组件的组合。

## 9. 续接时最重要的一句话

这个项目不是为了把潜点做成信息列表，而是为了让用户在浏览、选择和规划之间，慢慢进入一片更适合自己的海。

如果后续方案不能强化这一点，那通常就不是正确方向。
